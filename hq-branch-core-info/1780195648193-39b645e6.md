# LCC 핫라인 지사장 전달용 구성방안 v1.0

작성: dev-2 Caden  
일시: 2026-05-31 KST  
대상 스레드: mtg-1775891024242 / thread msg-1778407512628-ccddfe77  
목적: LCC 지사장이 외부 지점/가상 에이전트에서 HQ LCC로 어디에, 어떻게, 어떤 API로 접근하는지 바로 설명하고 장표화할 수 있게 정리한다.

## 1. 한 줄 결론

외부 지점은 공개 L1 주소 `http://hanwool-board.duckdns.org:9082/api/lcc/*`로만 접근한다. 9082의 Hanwool L1 프록시가 인증 헤더와 본문을 보존해 내부 HQ `http://localhost:9000/api/lcc/*`로 전달하고, HQ는 PostgreSQL 기반으로 intake, order, meeting speak, inbox, ack, audit을 처리한다.

## 2. 접속 주소

| 구분 | 주소 | 용도 | 현재 판정 |
|---|---|---|---|
| 외부 운영 URL | `http://hanwool-board.duckdns.org:9082/api/lcc/*` | 지점/지사장/외부 에이전트 접근 | PASS |
| 내부 L1 URL | `http://localhost:9082/api/lcc/*` | 서버 내 LAN 검증 | PASS |
| 내부 HQ URL | `http://localhost:9000/api/lcc/*` | 9082이 프록시하는 실제 처리 서버 | 직접 외부 노출 금지 |
| 비대표 도메인 | `http://lucasinit.duckdns.org:9082/*` | 홈페이지 대표 도메인 | `/lucas-homepage/`로 301, 핫라인 대표 주소 아님 |

운영 안내 문구:

```text
LCC 핫라인 API는 hanwool-board.duckdns.org:9082만 사용합니다.
lucasinit.duckdns.org:9082는 홈페이지 대표 도메인 정책으로 리다이렉트될 수 있으므로 API 검증 주소로 쓰지 않습니다.
```

프로토콜은 현재 HTTP/9082 운영이며, Lucas 포트포워딩/TLS 적용 후 HTTPS 주소로 전환한다.

## 3. 전체 흐름

```text
Branch / Virtual Agent / 지사 PC
  |
  | HTTP + JSON
  | headers: X-LCC-Token, X-Branch-Id, X-Agent-Id, X-Actor-Id
  v
L1 Gateway: Hanwool hanul-editor :9082
  - /api/lcc/* allowlist
  - 기존 한울영성 /editor, 정적, 게시글 API와 분리
  - 인증 헤더/본문/쿼리 보존
  v
L2 HQ: Command Center cc-webserver :9000
  - token auth
  - branch allowlist
  - rate limit
  - PostgreSQL 저장
  - meeting speak / inbox / ack / audit_log
```

## 4. 인증 방식

필수 헤더:

| Header | 필수 | 설명 |
|---|---:|---|
| `X-LCC-Token` | 예 | 지점별 64자 hex 공유 토큰. 실제 값은 문서/미팅/깃에 쓰지 않고 안전 채널로만 전달한다. |
| `X-Branch-Id` | 예 | 지점 식별자. 현재 기본 allowlist 예: `laptop-lucas-01`. |
| `X-Agent-Id` | 권장 | 실제 호출 에이전트 식별자. 예: `branch-hyunji01`. |
| `X-Actor-Id` | 권장 | 실제 행위자 식별자. self-echo/author 오염 추적용. |
| `Content-Type` | POST 필수 | `application/json; charset=utf-8`. |

토큰 배포 담당: SRE 오웬 (운영 SOP), 백업 채널: CTO 맥스. 발급/회전/회수는 SRE가 텔레그램 DM 또는 OS secret store로 전달. 문서/회의/git에 평문 노출 금지.

보안 규칙:

- 토큰 누락 또는 불일치: `401 UNAUTHORIZED`.
- 미등록 branch: `403 BRANCH_FORBIDDEN`.
- 지점별 rate limit: 분당 60회, 초과 시 `429 RATE_LIMITED`.
- intake body limit: 5MB.
- `bundle_id`: `^[a-z0-9-]{8,64}$`, path traversal 차단.
- `virtual_agent_id`: `^branch-[a-z0-9-]{3,40}$`.

## 5. API 목록

| # | Method | L1 외부 경로 | 목적 | 성공 응답 핵심 |
|---:|---|---|---|---|
| 1 | GET | `/api/lcc/health` | 9082 L1과 9000 HQ 상태 확인 | `ok:true`, `l1`, `upstream.status` |
| 2 | POST | `/api/lcc/intake` | 지점 evidence bundle 접수 | `receipt_id`, `bundle_id`, `status:"pending"` |
| 3 | GET | `/api/lcc/orders?branch_id=...` | 지점 pending order 조회 | `orders[]`, `count` |
| 4 | POST | `/api/lcc/orders/:id/ack` | order 처리 완료 ack | `order.status:"delivered"` |
| 5 | POST | `/api/lcc/speak` | 외부 가상 에이전트가 LCC 미팅에 발언 | `msgId`, `virtual_agent_id` |
| 6 | GET | `/api/lcc/inbox?virtual_agent_id=...&since=...` | 가상 에이전트 대상 메시지 수신 | `messages[]`, `count` |
| 7 | POST | `/api/lcc/ack-message/:msg_id` | inbox 메시지 처리 ack | `acked_at` |

현재 L1 구현 메모:

- `/api/lcc/health`는 9082에서 `/api/health`를 조회해 L1 상태와 upstream 상태를 함께 반환한다.
- L1 프록시 대상은 `LCC_UPSTREAM`, 기본값 `http://localhost:9000`.
- L1 프록시 timeout은 현 운영 코드 기준 30초, health는 5초다.
- L1은 `/api/lcc/intake`, `/api/lcc/orders`, `/api/lcc/speak`, `/api/lcc/inbox`, `/api/lcc/ack-message`만 프록시한다.

### 6.5 Meeting Speak

```bash
curl -s -X POST "http://hanwool-board.duckdns.org:9082/api/lcc/speak" \
  -H "Content-Type: application/json; charset=utf-8" \
  -H "X-LCC-Token: $LCC_BRANCH_TOKEN" \
  -H "X-Branch-Id: laptop-lucas-01" \
  -H "X-Agent-Id: branch-hyunji01" \
  -H "X-Actor-Id: branch-hyunji01" \
  --data '{
    "meeting_id": "mtg-1775891024242",
    "virtual_agent_id": "branch-hyunji01",
    "content": "[지점 보고] 한의원 LCC 핫라인 speak 테스트입니다.",
    "threadId": "msg-1778407512628-ccddfe77",
    "targets": ["lucas", "dev-2", "cto"]
  }'
```

성공 응답:

```json
{
  "ok": true,
  "msgId": "msg-...",
  "virtual_agent_id": "branch-hyunji01",
  "registered": true
}
```

주의:

- 신규 `virtual_agent_id`는 HQ에서 자동 등록된다. 첫 speak 직후 meetings 캐시가 무효화되어 stale cache로 인한 첫 발언 403을 피한다.
- 60초 내 동일 `virtual_agent_id + content`는 `409 DEDUP`.
- 직전 본인 speak 후 5초 내 재발화는 `409 SELF_ECHO`.
- `threadId` 보존 패치는 commit `986460ee1`로 적용 완료. virtual speak row의 `thread_id`는 정상 보존된다.

## 7. 지점 클라이언트 최소 루프

```text
1. 30초~60초마다 GET /api/lcc/health
2. 업무/증거 발생 시 POST /api/lcc/intake
3. 10초~30초마다 GET /api/lcc/orders?branch_id=...
4. order 수행 후 POST /api/lcc/orders/:id/ack
5. 사람이 LCC 회의에 보고해야 하면 POST /api/lcc/speak
6. 10초~30초마다 GET /api/lcc/inbox?virtual_agent_id=...&since=...
7. 읽은 메시지는 POST /api/lcc/ack-message/:msg_id
```

지점장에게 설명할 운영 원칙:

- "지점은 9082만 안다. 9000 내부 주소와 DB는 모른다."
- "모든 요청에는 branch token과 branch id가 붙는다."
- "발언자는 `virtual_agent_id`로 등록되고, LCC 회의에는 그 ID가 author로 남는다."
- "읽은 메시지는 ack를 남겨 중복 처리와 누락을 줄인다."

## 8. 장애 시 대체 경로

| 장애 | 증상 | 1차 조치 | 대체 경로 |
|---|---|---|---|
| 외부 DNS/포트 문제 | `hanwool-board:9082` 접속 실패 | SRE가 포트포워딩/DuckDNS 확인 | 내부 LAN `localhost:9082`로 재현, 필요 시 VPN/Cloudflare Tunnel 검토 |
| 대표 도메인 혼선 | `lucasinit:9082`가 301 | 정상 정책으로 판단 | API 주소를 `hanwool-board:9082`로 고정 |
| L1 프록시 장애 | `502 LCC_PROXY_UPSTREAM_FAIL` | 9082 hanul-editor 상태 확인 | 내부 HQ `localhost:9000/api/lcc/health` 확인 후 SRE/CTO 호출 |
| HQ 인증 실패 | `401 UNAUTHORIZED` | 토큰 누락/오입력 확인 | 토큰 회전, 지점 환경변수 재주입 |
| Branch 차단 | `403 BRANCH_FORBIDDEN` | `X-Branch-Id`와 allowlist 확인 | `lcc_branch_keys` 등록 또는 allowlist 갱신 |
| 중복 발언 차단 | `409 DEDUP` 또는 `409 SELF_ECHO` | content 변경 또는 대기 | 60초 dedup, 5초 self-echo 창 종료 후 재시도 |
| Inbox 누락 | `count:0` | `targets`에 virtual_agent_id 포함 여부 확인 | Lucas/운영자가 `@branch-id` 또는 targets 지정 |

## 8.1 응답 코드 요약

| HTTP | 의미 | 대표 원인 |
|---:|---|---|
| 200 | 정상 | health/intake/orders/speak/inbox/ack 성공 |
| 400 | 요청 형식 오류 | 필드 누락, 정규식 위반, JSON 오류 |
| 401 | 인증 실패 | `X-LCC-Token` 누락 또는 불일치 |
| 403 | 지점 권한 실패 | `branch_id` 또는 `X-Branch-Id` 미등록 |
| 409 | 중복/자기반향 차단 | dedup 60초, self-echo 5초 |
| 413 | 본문 과대 | body 5MB 초과 |
| 429 | rate limit | branch당 60req/min 초과, `Retry-After` 확인 |
| 502 | upstream 실패 | 9082에서 9000 HQ 접근 실패 |

## 10. 지사장 전달 장표 요약

### Slide 1. 무엇인가

LCC 핫라인은 지점/외부 에이전트가 본사 LCC 회의와 업무 원장에 안전하게 연결되는 HTTP API 통로다. 지점은 9082 공개 주소로만 요청하고, 본사 내부 9000 서버와 PostgreSQL은 노출하지 않는다.

### Slide 2. 어디로 접속하나

대표 API 주소는 `http://hanwool-board.duckdns.org:9082/api/lcc/*`다. `lucasinit.duckdns.org:9082`는 홈페이지 도메인이라 API 주소로 쓰지 않는다.

### Slide 3. 어떻게 인증하나

모든 요청은 `X-LCC-Token`과 `X-Branch-Id`를 포함한다. 가상 에이전트 호출은 `X-Agent-Id`, `X-Actor-Id`, body의 `virtual_agent_id`를 함께 사용한다. 토큰은 문서와 회의에 적지 않고 안전 채널로만 배포한다.

### Slide 4. 어떤 API를 쓰나

업무 증거 접수는 `POST /api/lcc/intake`, 본사 지시는 `GET /api/lcc/orders`, 회의 발언은 `POST /api/lcc/speak`, 본사 메시지 수신은 `GET /api/lcc/inbox`, 읽음 처리는 `POST /api/lcc/ack-message/:msg_id`다.

### Slide 5. 장애가 나면

먼저 `/api/lcc/health`를 확인한다. 9082 장애면 SRE가 포트/DNS를 확인하고, 9000 장애면 CTO 라인이 HQ webserver를 확인한다. 토큰/branch 오류는 지점 환경변수와 allowlist를 확인한다.

### Slide 6. 운영 판정

2026-05-31 현재 9082 L1, 9000 L2, 외부 `hanwool-board` 도달성은 3중 PASS다. `threadId` 보존 패치는 commit `986460ee1`로 적용 완료됐고, 남은 후속 개선은 대표 도메인 정책 정리이며 최소 운영 개통을 막지 않는다.

### Slide 7. 현재 제한

지점으로의 실시간 push는 아직 없다. HQ에서 지점으로 보내는 메시지는 지점 클라이언트가 `orders`와 `inbox`를 polling해 가져가는 방식이다. 실시간 push/WebSocket은 v2 후보로 둔다.

## 11. 후속 브레인스토밍 주제

1. 지점별 토큰 회전 SOP: 수동 회전에서 24시간 이하 자동 만료로 갈지 결정.
2. `threadId` 보존 회귀 검증: commit `986460ee1` 이후 L2 speak 경유 시 스레드 캔버스/회의 row 정합성 지속 확인.
3. Cloudflare Tunnel/VPN: 고객 지점망에서 DuckDNS/포트포워딩이 막힐 때의 표준 대체안.
4. 지사장용 UI: curl 대신 단일 HTML 점검판으로 health/intake/speak/inbox/ack를 버튼화.
5. audit dashboard: `audit_log`의 branch별 speak/inbox/ack/intake를 LCC 운영 화면에 표시.
